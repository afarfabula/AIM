# BisheMethod V2Stage：分阶段视觉 Token 压缩的设计理念与动机（Markdown / LaTeX 版）

## 1. 背景与问题定义

以 LLaVA-1.5 为代表的多模态大模型，通常将视觉编码器（vision tower）输出的视觉 patch token（如 `24×24=576` 个）经 projector 映射到语言模型隐空间，并与文本 token 拼接后输入 LLM 做自回归生成。评测任务（GQA/TextVQA/POPE 等）中，推理耗时与显存压力主要来自两部分：

1. 视觉侧：图像预处理与 vision tower 前向；
2. LLM 侧：长前缀（视觉 token + 文本 token）的注意力计算、KV cache 维护、以及生成阶段的解码。

**目标**：在尽量保持任务精度的前提下，将视觉 token 序列长度显著压缩到两位数（如 96），减少 LLM 侧前缀计算与生成开销，从而提升单卡推理吞吐。

---

## 2. 核心动机：为何需要“分阶段压缩”

若直接将 \(T_0=576\) 的视觉 token 一步压缩到两位数 \(T^\*\approx 10^2\)，常见风险是不可逆的信息损失：

- 关键局部证据（小物体、文字区域、关系证据）过早被合并或消失；
- 合并策略对噪声敏感，早期错误会在后续阶段放大且难以纠正；
- 仅凭视觉 token 自相似的单一度量，可能偏好背景纹理或全局区域，忽略与任务相关的局部细节。

因此 **BisheMethod V2Stage** 采用“渐进式、多阶段压缩”，将一次性困难优化转化为多个稳定的中间表示学习/构造步骤，降低早期不可逆误差。

---

## 3. 方法概述

BisheMethod V2Stage 是一种**阶段感知（stage-aware）的视觉 token 软合并（soft merge）框架**，核心包含：

1. **多阶段预算调度（Progressive Budget Schedule）**
2. **阶段化度量（Stage-wise Metric）**
3. **软保护机制（Soft Anchor Protection）**

其高层思路是：**阶段 1 更强调结构/语义保真，阶段 2/3 更强调稳定压缩与鲁棒性**。

---

## 4. 设计细节与理由

### 4.1 多阶段预算调度（渐进压缩）

设初始视觉 token 数为 \(T_0\)，目标 token 数为 \(T^\*\)，采用分阶段调度：

\[
T_0 \rightarrow T_1 \rightarrow T_2 \rightarrow \cdots \rightarrow T_S = T^\*
\]

一个典型实例为：

- Stage 1：\(576 \rightarrow 304\)
- Stage 2：\(304 \rightarrow 156\)
- Stage 3：\(156 \rightarrow 96\)

这种“粗到细”的压缩可降低一步到位带来的灾难性合并风险，并使每一阶段的合并决策更可控。

---

### 4.2 为什么用“软合并（merge）”而不是“硬裁剪（prune）”

硬裁剪直接丢弃 token，信息不可恢复；软合并通过将多个 token 的信息聚合到保留 token 中，使得最终 token **更像信息摘要**，对 VQA/关系推理中“组合证据”（对象+属性+关系）的保留更友好。

可将阶段 \(s\) 视作从 \(X^{(s)}\in\mathbb{R}^{T_s\times d}\) 构造一个更短的表示 \(X^{(s+1)}\in\mathbb{R}^{T_{s+1}\times d}\)。

---

### 4.3 阶段化度量：Stage 1 更“丰富”，后续更“稳定”

阶段 1 的目标是形成一个仍保留语义与空间结构的中间表示，因此引入更丰富的相似度/匹配度量；后续阶段预算更紧，错误代价更大，因此回退到更稳定、更鲁棒的度量。

可将阶段 1 的度量写成混合形式：

\[
M^{(1)} = \alpha \cdot M_{qk}(X) + \beta \cdot M_{pos}(X) + (1-\alpha-\beta)\cdot M_{base}(X)
\]

- \(M_{qk}\)：来自注意力投影空间（如 \(q/k\) 相关表示）的相似度项，动机是更贴近后续 LLM 交互所依赖的表征；
- \(M_{pos}\)：轻量空间位置项，用于抑制空间塌缩与过度向背景集中；
- \(M_{base}\)：更稳定的基础相似度项（例如视觉 token embedding 的余弦相似）。

阶段 2/3 逐步降低 \(\alpha,\beta\)，趋向更稳定的 \(M_{base}\)，以降低极低 token 数下的方差与脆弱性。

---

### 4.4 软保护机制（Soft Anchor Protection）

为避免关键结构性 token（例如代表空间覆盖或潜在关键区域的 token）在早期被过快合并，V2Stage 对“受保护集合” \(\mathcal{P}\) 施加软惩罚（penalty）而非硬约束：

\[
score(i) \leftarrow
\begin{cases}
score(i) - \lambda, & i \in \mathcal{P} \\
score(i), & i \notin \mathcal{P}
\end{cases}
\]

- \(\lambda>0\) 为软惩罚强度；
- 软惩罚不会绝对禁止合并，只是降低其被合并的概率；
- 相比硬保留锚点，软保护在极紧预算下不会强行侵占容量，更适配不同数据集差异。

---

## 5. 算法过程（形式化简述）

设阶段 \(s\) 的 token 为 \(X^{(s)}\in\mathbb{R}^{T_s\times d}\)，目标为 \(T_{s+1}<T_s\)。定义合并数量：

\[
r_s = T_s - T_{s+1}
\]

基于度量 \(M^{(s)}\) 计算匹配/合并策略，并结合软保护（对 \(\mathcal{P}\) 做 penalty），执行软匹配合并得到：

\[
X^{(s+1)} = \mathrm{SoftMerge}\left(X^{(s)}, M^{(s)}, r_s, \lambda, \mathcal{P}\right)
\]

重复直到 \(T_S=T^\*\)。

---

## 6. 与基线思路的关系（定位）

- 与 AIM 类方法的共同点：强调“合并”而非“裁剪”，用信息聚合构造短序列表示；
- 与单阶段压缩的区别：V2Stage 采用**渐进调度 + 阶段化度量**，降低不可逆误差累积；
- 与纯空间锚点硬保留的区别：V2Stage 用**软保护**注入先验，减少对固定锚点的强依赖并提升跨数据集可迁移性。

---

## 7. 工程实现要点（便于复现）

- 插入位置：在进入 LLM 之前完成视觉 token 压缩；因此不改变 LLM 的采样/解码机制，改变的是进入 LLM 的视觉前缀表示与长度。
- 额外计算：主要来自 token 相似度/匹配计算（常见为 \(O(T^2)\) 或近似版本），但由于 token 数在第一阶段后迅速降低，整体可控；同时 LLM 侧因前缀变短带来的注意力计算下降通常能抵消部分代价。
- 超参数语义（对应实现中的常见字段）：
  - `merge_steps`：多阶段预算调度；
  - `qk_mix`：阶段 1 中 \(M_{qk}\) 的权重 \(\alpha\)；
  - `pos_mix`：阶段 1 中 \(M_{pos}\) 的权重 \(\beta\)；
  - `penalty`：软保护强度 \(\lambda\)。

---

## 8. 局限性与后续方向

- 当前主要利用视觉侧内部信息（及其投影空间）进行压缩，未显式引入文本问题作为 query，因此可能保留部分与问题无关的视觉信息。
- 可扩展为 **text-query-aware**：以文本 token 的聚合向量作为 query，引导视觉 token 的保留/合并，使在两位数 token 预算下仍能最大化任务相关证据的保真度。

---

## 9. 一句话摘要（给文档 Agent）

BisheMethod V2Stage 是一种面向多模态推理加速的“分阶段渐进式视觉 token 软合并”方法：通过渐进预算调度、阶段 1 的语义增强度量（如 q/k + 轻位置项）与后续阶段的稳定回退，以及软保护机制，在将 576 视觉 token 压缩到两位数（如 96）的同时尽量维持 VQA 任务精度，并且不改变 LLM 的生成行为，仅改变进入 LLM 的视觉前缀表征与长度。
